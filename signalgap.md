**Verdict**
- Yes, BuzzU is **actually using a Signal-like protocol** for chat payload encryption.
- But it is **not a production-robust Signal implementation** yet compared to canonical libs.
- Practical rating today: **medium security for passive attackers, weak against active MITM / lifecycle failures**.

**Where it is used in BuzzU**
- Key exchange and session setup are wired in chat flow via `PublishKeys` → `RequestKeys` → `KeysResponse` → `SignalHandshake` in [ChatArea.tsx](file:///c:/Users/will/Desktop/BuzzU/apps/web/src/components/Chat/ChatArea.tsx#L2625-L2890).
- Message send path enforces encryption and blocks plaintext fallback in [ChatArea.tsx](file:///c:/Users/will/Desktop/BuzzU/apps/web/src/components/Chat/ChatArea.tsx#L1933-L2038).
- WASM protocol object is created and sessions managed in [useCrypto.ts](file:///c:/Users/will/Desktop/BuzzU/apps/web/src/hooks/useCrypto.ts#L7-L63).
- Signaling worker forwards Signal-related messages in [lib.rs](file:///c:/Users/will/Desktop/BuzzU/apps/signaling-worker/src/lib.rs#L682-L730).

**Critical findings (robustness gaps)**
- **One-time prekey is not actually a DH one-time prekey**:
  - `onetime_prekey` carries Ed25519 verifying key bytes, not an X25519 OTPK in [signal.rs](file:///c:/Users/will/Desktop/BuzzU/packages/wasm/src/crypto/signal.rs#L124-L129).
  - `x3dh_initiate(..., None)` disables DH4 in [signal.rs](file:///c:/Users/will/Desktop/BuzzU/packages/wasm/src/crypto/signal.rs#L191-L197).
- **Out-of-order resilience is incomplete**:
  - Skipped message keys are not persisted; comment admits full implementation missing in [signal.rs](file:///c:/Users/will/Desktop/BuzzU/packages/wasm/src/crypto/signal.rs#L420-L427).
- **No persistent identity trust model (TOFU/pinning)**:
  - Sessions are intentionally ephemeral and not persisted in [useCrypto.ts](file:///c:/Users/will/Desktop/BuzzU/apps/web/src/hooks/useCrypto.ts#L11-L15).
- **PublishKeys is broadcast to room peers**:
  - `PublishKeys` is broadcast to all peers in [lib.rs](file:///c:/Users/will/Desktop/BuzzU/apps/signaling-worker/src/lib.rs#L682-L700), widening exposure in non-2p rooms.
- **Encrypted signaling transport exists but is unused**:
  - `useEncryptedSignaling` exists, but no other file imports it (only definition file matches).
  - Hook file: [useEncryptedSignaling.ts](file:///c:/Users/will/Desktop/BuzzU/apps/web/src/hooks/useEncryptedSignaling.ts#L57-L209).

**GitHub MCP benchmark (production patterns)**
- I compared against Signal’s own Rust implementation:
  - `signalapp/libsignal-protocol-rust` storage traits require identity/prekey/signed-prekey/session stores:  
    https://github.com/signalapp/libsignal-protocol-rust/blob/master/src/storage/traits.rs
  - Session cipher enforces trusted identity checks, prekey consumption, previous-session handling, and forward-jump controls:  
    https://github.com/signalapp/libsignal-protocol-rust/blob/master/src/session_cipher.rs
- Key difference: BuzzU currently implements core ratchet mechanics, but misses the full store/trust/prekey lifecycle hardening used in production-grade Signal stacks.

**Bottom line**
- **Is Signal protocol really used?** Yes, for chat message crypto/session handshake.
- **Is this robust like production Signal?** Not yet. It is a custom Signal-inspired implementation with important missing hardening layers.
